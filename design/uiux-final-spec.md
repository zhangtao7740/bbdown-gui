# BBDown GUI 最终 UI/UX 设计稿

## 定位

BBDown GUI 是一个桌面下载工具，不是展示型网站。第一屏必须直接可用：粘贴链接、解析、选择下载内容、开始下载。视觉目标是“安静、清楚、能长期盯着用”，避免大面积营销式 hero、过度渐变和装饰。

## 主方案保留点

- 保留左侧主导航：下载、队列、历史、设置、关于。
- 保留顶部 URL / BV 输入区，并让“解析”和“开始下载”成为一组明确动作。
- 保留中间视频预览 + 下载选项的主工作区。
- 保留右侧状态栏：工具状态、账号状态、最近任务。
- 保留底部状态栏：就绪/解析中/下载中，以及当前选择数量。
- 保留浅色专业桌面工具气质。

## 需要调整点

- 主图里右侧最近任务较像“小卡片列表”，实现时建议压缩高度，避免和队列页重复。
- 下载选项不要全部塞成一张长表；按“画质 / 内容 / 保存 / 高级”分区。
- “API 模式”属于高级配置，默认折叠或放到下载选项底部。
- “仅音频”和“仅视频”是互斥模式，应做 segmented control，而不是两个可同时勾选的 checkbox。
- Fluent UI 可不保留。最终视觉应由 design tokens 控制，而不是由组件库默认样式决定。

## 信息架构

1. 下载
   - 链接输入
   - 解析结果预览
   - 分 P 选择
   - 下载选项
   - 工具/账号/最近任务状态
2. 队列
   - 运行中、等待中、暂停、失败、完成
   - 单任务操作：暂停、继续、取消、打开目录、查看日志
   - 批量操作：全部暂停、清理完成、重试失败
3. 历史
   - 统计概览
   - 搜索与筛选
   - 下载产物列表
   - 后处理：重命名、移动、重新定位、转码/封装
4. 设置
   - 账号 / Cookie
   - 工具路径
   - 下载默认值
   - 后处理默认值
   - 外观和系统
5. 关于
   - 版本
   - 工具版本
   - 项目链接
   - 许可证

## 页面设计

### 下载页

布局：左导航 180px，主区 `minmax(560px, 1fr)`，右侧状态栏 320px。小屏时右侧状态栏下移。

顶部输入区：
- 输入框 placeholder：`粘贴 Bilibili 链接 / BV 号`
- 次按钮：解析
- 主按钮：开始下载
- 解析中：输入框禁用，解析按钮 loading
- 未解析：开始下载禁用

视频预览：
- 封面 168x104
- 标题两行截断
- 元信息：UP 主、BV、发布时间、时长、分区
- 分 P 使用可滚动列表或 chips，超过 8 个时折叠

下载选项：
- 画质：自动、8K、4K、1080P 高码、1080P、720P
- 内容：完整视频、仅音频、仅视频
- 附加资源：字幕、弹幕、封面
- 格式：字幕语言、弹幕格式、音频质量、视频编码
- 性能：aria2c、并发数
- 保存：路径输入 + 浏览
- 高级：API 模式、跳过 AI 字幕、命名模板

右侧状态栏：
- 工具状态：BBDown、FFmpeg、aria2c
- 账号状态：未登录 / 已登录 / Cookie 即将过期
- 最近任务：最多 4 条，点击进入队列或历史

### 队列页

顶部：
- 标题：任务队列
- 统计 chips：运行中、等待、失败
- 操作：全部暂停、全部继续、清理完成

任务卡：
- 左侧封面
- 中部标题、BV、UP、保存目录
- 右侧状态 badge 和操作图标
- 进度条显示下载阶段：解析、下载、合并、后处理
- 日志默认折叠，展开后使用等宽字体

### 历史页

顶部统计：
- 总任务、完成、失败、产物、缺失文件、占用空间

列表：
- 默认按完成时间倒序
- 搜索标题 / BV / UP / 文件名
- 筛选状态、产物类型、日期
- 产物行显示：类型、文件名、媒体信息、大小、存在状态、操作

后处理：
- 视频：Remux MP4、H.264 + AAC MP4、MKV
- 音频：MP3、M4A/AAC、FLAC
- 操作要在历史产物上执行，不放回下载页。

### 设置页

分组使用 full-width section，不要做卡片套卡片。

- 账号：扫码登录、退出、刷新状态。提示 BBDown.data 是本地凭据文件。
- 工具路径：自动检测 + 手动选择 + 重新检测。
- 下载默认值：默认目录、并发数、默认清晰度、默认附加资源。
- 后处理：默认容器、音频转码、保留源文件。
- 外观：浅色、深色、跟随系统。
- 系统：通知、启动时检查更新、最小化到托盘。

### 关于页

- 应用图标和版本。
- BBDown GUI、Electron、BBDown、FFmpeg、aria2c 版本。
- GitHub、许可证、问题反馈。
- “复制诊断信息”按钮，便于用户报错。

## 视觉规范

颜色：
- 背景：`#F7F9FC`
- 面板：`#FFFFFF`
- 侧栏：`#F4F7FB`
- 主色：`#0F6CBD`
- 主色 hover：`#0A5EB8`
- 成功：`#1FA463`
- 警告：`#C17700`
- 错误：`#D13438`
- 主文字：`#101828`
- 次文字：`#667085`
- 边框：`#D8E0EA`

圆角：
- 窗口：10px
- 卡片/面板：8px
- 输入框/按钮：7px
- chips：6px

尺寸：
- 标题栏：Windows 32px，macOS 38px
- 顶部输入区：96px 高
- 主按钮：40px 高
- 侧栏项：44px 高
- 状态栏：32px 高

状态色：
- 下载中：蓝色
- 处理中：紫蓝色，可用 `#5B5FC7`
- 完成：绿色
- 失败：红色
- 暂停/等待：灰色

## 明确不做范围

- 不做 v1 到 v2 数据迁移，采用 breaking change 策略。
- 不做新手引导。
- 不做性能指标体系。
- 不做拖拽链接或文件。
- 不做自动更新。
- 不做 i18n。
- 不做 A11y 专项规范。

## 最终结论

主设计图的布局方向值得保留，Fluent UI 不是必须保留。最终方案应是 Electron 原生壳 + React 业务组件 + 自有 design tokens + Radix/lucide 辅助组件。这样能在 Windows 上保持工具软件清晰度，在 macOS 上减少 Windows 化痕迹，并且实现复杂下载状态、历史产物和后处理流程时更可控。
